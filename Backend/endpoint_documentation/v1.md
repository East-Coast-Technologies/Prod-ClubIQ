# ClubIQ Backend API v1

This document describes the production-ready v1 backend API surface for ClubIQ.

v1 is intentionally a single-club backend release. The backend owns the active club context through environment configuration, not frontend-provided `club_id`.

## Base URL

```text
/api/v1
```

## Authentication

All protected endpoints require a valid Clerk session token.

```http
Authorization: Bearer <Clerk session token>
Content-Type: application/json
```

The backend verifies the Clerk token and resolves the current synced user from the token.

The frontend must not send `user_id` to identify the logged-in user.

## v1 Production Rules

v1 uses one configured club.

The active club is selected by:

```env
SINGLE_CLUB_NAME=<club-name>
```

The frontend must not send `club_id` for v1 activities or members.

Legacy APIs can be disabled in production:

```env
EXPOSE_LEGACY_API=false
```

The scheduler management API should stay disabled in production:

```env
SCHEDULER_API_ENABLED=false
```

## v1 Route Surface

When production config is enabled with:

```env
EXPOSE_LEGACY_API=false
SCHEDULER_API_ENABLED=false
```

the backend exposes the v1 surface:

```text
/api/v1/activities
/api/v1/activities/
/api/v1/auth/me
/api/v1/auth/sync/
/api/v1/auth/test/
/api/v1/club/
/api/v1/health
/api/v1/health/
/api/v1/members
/api/v1/members/
/api/v1/members/<member_id>
/api/v1/ratings/<rating_id>
/api/v1/ratings/<task_id>/
/api/v1/ratings/user/<user_id>/
/static/<path:filename>
```

and keeps Docker/backend liveness routes available:

```text
/backend-health
/backend-health/
/api/backend-health
/api/backend-health/
```

## v1 Endpoints

| Area | Method | Endpoint | Description |
|---|---|---|---|
| Health | GET | `/api/v1/health/` | Confirms backend is running |
| Auth | POST | `/api/v1/auth/sync/` | Syncs the signed-in Clerk user into the local DB |
| Auth | GET | `/api/v1/auth/me` | Returns current user, active club, membership, and access context |
| Auth | GET | `/api/v1/auth/test/` | Protected auth test endpoint |
| Club | GET | `/api/v1/club/` | Returns the configured single active club |
| Activities | GET | `/api/v1/activities/` | Lists activities for the configured club |
| Activities | POST | `/api/v1/activities/` | Creates an activity in the configured club |
| Members | GET | `/api/v1/members/` | Lists members in the configured club |
| Members | POST | `/api/v1/members/` | Creates a membership in the configured club |
| Members | GET | `/api/v1/members/<member_id>` | Gets one member from the configured club |
| Members | PUT | `/api/v1/members/<member_id>` | Updates one member from the configured club |
| Members | DELETE | `/api/v1/members/<member_id>` | Deletes one member from the configured club |
| Ratings | GET | `/api/v1/ratings/<task_id>/` | Lists ratings for a task |
| Ratings | POST | `/api/v1/ratings/<task_id>/` | Creates a rating for a completed task |
| Ratings | GET | `/api/v1/ratings/user/<user_id>/` | Lists ratings for a user |
| Ratings | DELETE | `/api/v1/ratings/<rating_id>` | Deletes a rating |

## Not Exposed in v1

These routes are intentionally not part of the v1 production API:

```text
/api/v1/clubs
/api/v1/clubs/<club_id>
/api/v1/invitations
/api/v1/invitations/<invite_id>
/api/v1/invitations/<token>/accept
/api/v1/activities/<club_id>/
/api/v1/members/?club_id=<club_id>
/api/v1/auth/me/<user_id>/
```

## Invitation Flow in v1

v1 does not expose local backend invitation endpoints under `/api/v1`.

Invitation and sign-up access should be handled by Clerk.

The Flask backend only verifies Clerk authentication, syncs authenticated users, and checks whether the synced user belongs to the configured club.

## Access Control

A synced user can access v1 club data only if they are allowed inside the configured club.

Allowed users:

```text
- member of the configured club
- creator of the configured club
- admin
- super_user
```

Blocked users:

```text
- synced but not a member of the configured club
- unsynced Clerk user
```

Blocked users receive `403`.
